# fdb — 微型键值存储引擎 开发笔记

---

## 一、架构设计

对于 fdb 键值管理器这个项目，我们需要先确认其功能。通过把一条条数据按书页般存储起来，并配上"封面"（标注了关键信息），我们便可以通过简单指令，实现对这些数据的写入、读取、遍历、删除、刷新。我们的 main.c 就用来执行具体的命令操作，不过这和交互式的 mini-shell 不同——参数由 shell 拆分成 `argc`/`argv` 传入 `main()`，不像 mini-shell 需要自己 `readline` + `strtok` 逐行解析输入。因此，在主程序里，只需要对输入的命令进行检测就可以。

> **引申：CLI vs REPL**
> —— `fdb` 是一次性命令行工具（CLI，Command-Line Interface），`mini-shell` 是交互式 REPL（Read-Eval-Print Loop）。CLI 的特点是"启动 → 执行 → 退出"，无状态；REPL 的特点是"循环等待输入 → 解析 → 执行 → 等待下一次输入"。Unix 核心工具（`ls`、`cp`、`grep`）几乎全是 CLI 模式。

那么，对于我们所希望的"书页"，我们可以展开思考，所谓书页，必然有索引，而索引对应的就是数组，但我们知道，单一的数组是很难进行数据的插入和删除的，而我们既然要做成书页，就可以做到随时删改，这一方面，链表的功能就比较强大，有着一套自己的增删改查规则，不过其也缺乏对大量数据的管理能力，既要数量，也要方便，两者联系，我们就可以自然采用数组 + 链表的方式来构成我们的索引 + 数据模式，而对于如此多的书页来说，如何让数据分布的均匀呢？这就引入了哈希表和哈希函数，这个函数可以为一个字符串计算出对应的索引值，并且可以根据你给的数字，让不同字符串的索引尽量均匀分布在一个范围。索引的方案就此完成。

> **术语：链式哈希表（Chained Hash Table）**
> —— 你描述的"数组 + 链表"结构，在内核中称为链式哈希表（也叫拉链法 / Separate Chaining）。`buckets` 是固定大小的指针数组，每个桶通过链表解决哈希碰撞（Hash Collision）。Linux 内核中到处是这种结构——`struct hlist_head`（哈希链表头）和 `struct hlist_node`（链表节点）就是它的标准实现。Phase 2 的 `cbits` 项目中会深入学习 `container_of` 和内核链表。

与索引对应的就是数据的操作方法，对 record 和 header 的增删改查，我们也需要一个额外的文件来完成，把这个文件和索引文件联合起来，对他们进行调用，完成我们期望的几个命令函数的构建，就是我们的对外 API 文件，我们这里也不难看出，fdb 的命令是一种增删改查的形式，而我们的增删改查，是在数据和索引两个层面进行的。

> **术语：分层架构（Layered Architecture）**
> —— 将系统拆分为多层，每层只专注一件事，上层调用下层接口，下层不感知上层逻辑。fdb 的三层结构（record → index → fdb）遵循了 **关注点分离（Separation of Concerns）** 原则，和 Linux 内核的 VFS 分层（VFS → 文件系统 → 块设备驱动）思想一致。

综上所述，我们便设计好了四个文件：`record.c`（磁盘读写层）、`index.c`（内存索引层）、`fdb.c`（业务逻辑层）、`main.c`（CLI 入口层）。

```
CLI (main.c)      ← 解析命令行，调业务层
   ↓
业务 (fdb.c)      ← get/set/delete/list/compact 逻辑
   ↓           ↓
磁盘 (record.c)   索引 (index.c)
```

## 二、record.c — 磁盘读写层

```c
/* record.c — 磁盘读写层
 * 只关心"给定编号N，把字节写进去/读出来"，不懂key/value含义
 */
```

这一部分，是对 fdb 数据管理的重要部分，它是一个底层文件，我们 fdb 文件的创建、修改、读取都要用到它。既然是一个文件，逃不过创建、读写、修改这几个过程（至于删除，可以直接使用写操作进行废弃）。

那么首先就是创建，我们只会在第一次运行时进行创建，我们会对初始创建的 fdb 文件进行 header 和 record 的写入，具体控制写入位置就使用 lseek 函数，操控光标，确认"落笔"的位置。在完成初始创建后，尽管此时的 record 只使用 0 来填充的空白书页，但有书皮有书页，已经齐全。

对于读和写，我们选择把 header 与 record 分开，header 毕竟简单，因为作为书皮，位置必定在开头，一次读写，错误检测，就可以完成功能。

record 的读写，又有不同的讨论，对于比较简单的读取，我们的 lseek 也要有不同，我们通过 lseek 移动到指定位置，其偏移公式为 header 的大小 + N × record 的大小。

record 的写入，我们分为两种情况，改单位置和改多位置，后者要考虑安全问题，比如突然断电之类，要采用原子写入的方法。所以前者和读取是一样的路数，而后者就要更复杂些。为了保证原子性，我们可以采用 rename 函数。我们知道，对于一个文件，其有目录项（文件名）、inode（门牌号）、数据块（仓库）这三个组成，rename 函数需要两个参数，我们可以把他们当做"保留 inode 的文件"和"保留目录项的文件"，我们通过把后者的目录项指向前者，这时候前者的目录项就被抛弃了，自然被删掉，而后者的 inode 没有了对应的目录项也会被我们清除掉，简单的指针操作，实现了一种覆盖（重命名）。整个过程不会有数据混淆的风险。因此，我们可以采用这个方法，先把源文件修改部分之外的数据原封不动的写在一个临时文件里，同时把修改部分填进去，在新文件写好后，再使用 fsync 来催促文件信息快速从页缓存中离开，刷到磁盘里，此时，再 rename，我们便完成了原子写入操作。

> **术语：原子性（Atomicity）、崩溃一致性（Crash Consistency）、页缓存（Page Cache）**
> —— 原子操作指"要么全做，要么全不做"，不存在中间状态。`rename` 的原子性由文件系统保证（通常依赖日志 / Journal）。崩溃一致性指系统断电重启后，数据不会出现逻辑矛盾（如"新数据记录在了旧页号"）。页缓存是内核在内存中维护的磁盘数据缓冲层——`write()` 返回时数据还在缓存中，`fsync()` 强制刷回物理存储，保证持久化（Durability）。

那么细节上，复写部分是值得注意的。我们先把临时文件的 header 填好，然后使用一个循环，保证能把 record 全部复写，我们可以读一步写一步，通过计数器 i 来操控完成一读一写操作后，再一次进循环时，lseek 能正好移动一个书页大小。此外，我会通过条件判断来对那个"改动页"进行定位。除了复写部分，还有一些 C 语言运用上的坑，比如要注意不要生成野指针，对于文件的 open 一定要慎重，路径必须正确，否则有删除源文件的风险，对于循环也要设置正确的退出条件，防止死循环。

## 三、index.c — 内存索引层（链式哈希表）

```c
/* index.c — 内存索引层（链式哈希表）
 * 只关心"key→编号"的映射，不懂文件格式
 */
```

index 作为内存索引层，我们已经知晓了其链式哈希表的主要结构，至于功能，就是要实现索引层的增删改查，总体上就是链表的运用。这里采用的哈希算法是 **djb2**（由 Daniel J. Bernstein 在 1991 年提出），核心公式为 `h = h * 33 + c`（写成位运算即 `((h << 5) + h) + c`），初始种子值 5381，最终对桶数量取模。本质是把字符串当作 33 进制数换算成十进制，再映射到 0 ~ size-1 的桶范围内。同一字符串计算出的桶编号是确定的（确定性），不同字符串的概率分布在各个桶上（均匀性）。

首先通过哈希函数可以获取字符串对应的"大"索引（所在的桶），每一个桶对应着一条链表，然后就是链表的增删改查基础操作。值得注意的是，我们在每次重启时构建索引的 build 操作中是需要用到 record 操作的，我们需要对 record 进行遍历，再针对性地进行 insert 操作。

> **引申：内存布局 —— 堆 vs 栈**
> —— `index_create` 中通过 `malloc` 在堆上分配 `index_t`、`buckets` 数组、以及每个 `entry`。这些分配的生命周期跨越整个函数调用链——从 `fdb_open` 到 `fdb_close` 才释放。而 `index_find` 中创建的 `p` 指针只是栈上的临时指针变量（指向堆上的 entry），函数返回即销毁，不涉及堆内存泄漏。

**index.c 踩坑记录：**
- `prev = p; p = p->next;` 写到 while 循环 `}` 外面，导致死循环 + 未初始化的指针被解引用。
- `sizeof(index_entry_t*)` 和 `size * sizeof(index_entry_t*)` 的一字之差，导致只分配了 8 字节（一个指针大小）而非整个桶数组。
- 删除链表节点时试图 `p->next = NULL` 后再 `free(p)`——完全多余，`free()` 不在乎被释放的节点 next 指向哪里。

## 四、fdb.c — 核心业务层

```c
/* fdb.c — 核心业务层
 * 把 record（磁盘读写）和 index（内存索引）组合成对外API
 */
```

fdb 是我们的核心业务层，在有了索引层操作 index 和数据层操作 record 后，我们只需要对他们进行组合，就可以构成我们的对外 API，逻辑简单但最重要，这是我们命令构建的核心过程。

首先我们每次发出命令，都需要打开（新建）文件，建立索引，这就是 open；而在命令完成后，也当然需要关闭文件的通道、释放空间、摧毁索引，这就是 close。

> **引申：资源管理模式 —— RAII 在 C 中的朴素实现**
> —— C++ 用构造/析构函数自动管理资源（RAII：Resource Acquisition Is Initialization）。C 里做不到自动，但遵循同样的模式：`fdb_open` 分配并初始化所有资源 → `fdb_close` 按逆序释放所有资源。调用者只需确保"打开后一定调关闭"，资源不会泄漏。fdb 这里还引入了 `fdb_t` 结构体来打包状态（fd + header + index），这是 **不透明指针（Opaque Pointer / Handle）** 模式的雏形——用户拿到的是一个不透明的 db 句柄，无需了解内部结构。

然后便是增添数据的操作 set，这一个命令比较复杂，我们增添数据，优先考虑空位和墓碑位（删除的数据），然后才会在尾端新增数据（这样我们必须要对 record_count 进行额外修改）。所以要对 record 进行遍历操作，由于这个操作势必会改动不止一处，我们必须采用原子写。此外，我们不仅要在数据层改动，索引层也不能忘记，所以还要以 insert 收尾。

删除命令，同样是先删文件，再删索引。get 命令是简单查找，list 命令是简单遍历。

最后的 compact 命令，这种大范围的改动，我们整体模仿原子写的操作，同样采用临时文件 + rename，在循环中按照读 → 判断 → 写的操作，清除墓碑，只保留有效 flag 和 0 flag，实现了 fdb 文件的"垃圾回收"。

> **术语：碎片整理 / Compaction、垃圾回收 / GC**
> —— 这里的 compact 本质上和 LevelDB 的 Compaction 思想一致——多条数据反复增删后，磁盘上产生碎片（墓碑 + 空闲槽混在有效数据中间），compact 将有效数据搬到新文件，丢弃无效数据，恢复空间连续性。在数据库领域这也叫 **垃圾回收（Garbage Collection, GC）**，通常通过 **写时复制（Copy-on-Write）** 或 **日志合并（Log-Structured Merge）** 实现。fdb 的做法属于前者——拷贝有效记录到新文件，原子替换。

**fdb.c 踩坑记录：**
- `fdb_t *db` 声明后没 malloc 直接 `db->fd = fd`——野指针，必崩。
- `fdb_compact` 中 `close(fd); db->fd = fd;` 顺序反了——已关闭的 fd 赋给 db->fd，后续所有磁盘操作全失败。
- `strncpy(rec.key, key, sizeof(rec.key))` 没留 `\0` 的位置——如果 key 长度正好等于 buffer 大小，字符串没有结尾符，后续 `strcmp` 和 `printf` 会越界输出。

## 五、总结

总的来说，这一次的项目，在架构实现的复杂性上远远不如 mini-shell，但这种功能更具体的项目，却是更考验我们对于功能背后的架构的挖掘能力。mini-shell 的各种操作很费脑，但是很容易总结出各个文件中的关系，而这个项目横跨了数据结构、Linux 文件操作，对于不成熟的学习者（比如我），实际上是更难独立完成的，而且这个文件涉及的错误检查要更多，我在编写中时常出现遗忘，这一点要注意。

**本次项目涉及的系统调用和技术栈：**

| 系统调用 / 函数 | 用途 | 所在层 |
|:---|------|:---:|
| `open/read/write/lseek/close` | 基础文件 IO | record |
| `stat` | 文件元信息 | record |
| `fsync` | 强制刷盘（持久化保证） | record |
| `rename` | 原子替换文件（原子性保证） | record |
| `snprintf` | 构造临时文件路径 | record/fdb |
| `malloc/free` | 堆内存管理 | index/fdb |
| `memset` | 内存清零初始化 | record/index/fdb |
| `strncpy/strcmp/strlen` | 字符串操作 | 全部 |
| `hash (djb2)` | 键到桶编号的确定映射 | index |
