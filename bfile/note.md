# bfile — 二进制文件解析器 开发笔记

---

## 一、项目定位

**功能一句话**：把二进制文件按类型定义翻译成可读的值。

**类型**：CLI 工具——一次命令一次执行，跑完退出。与 fdb 采用的位置参数模式不同，由于 bfile 命令的参数比较多，又不固定（没特殊要求，按理也不需要输入），这一次我们采用选项参数模式，这样没有次序要求，但需要我们自己对命令进行解析提取。

```
CLI vs 位置参数 vs 选项参数:
  fdb:  ./fdb test.db set name Alice        ← 位置参数，顺序固定
  bfile: ./bfile -f data.bin -s "..." -e big ← 选项参数，位置随意，可选可有默认值
```

> **引申：POSIX 选项约定**
> 短选项（`-f`）单横杠，长选项（`--file`）双横杠，布尔选项（`-v`）无值存在即生效。
> bfile 当前只实现了短选项，扩展长选项只需加 `strcmp(argv[i], "--file")` 镜像分支。

## 二、架构设计

```
main.c  — 选项参数解析 + 字段遍历 + 格式化输出
  │
  ├── parse.c — 把格式串 "uint32_t magic; char name[16]" 拆成 field_def_t 数组
  │
  └── read.c  — dispatch 表 + 11 种类型的读取函数 + 字节序转换
       └── bfile.h — 类型枚举 + 函数指针 typedef + 两套结构体 + extern 表声明
```

对 bfile 项目，其功能是输入指令，通过解析指令的信息，对指定文件进行数据类型的转换，读取所需要的数据。解析完命令后，程序了解到了需要读什么内容，但具体的读取，就需要一个 read 程序。

**关键设计决策：如何处理 11 种类型？**

我们读取的文件都是二进制，不过输出不同，这需要每一种类型都有对应的读取函数。如此庞大的函数群，彼此又相似，我们需要一种简便的方式来管理，来代替 if-else。

```
不用 dispatch 表:
  if (type == TYPE_UINT8)       read_uint8(...)
  else if (type == TYPE_UINT16) read_uint16(...)
  else if (type == TYPE_UINT32) read_uint32(...)
  // 新增类型 → 再加一个 else if → 代码膨胀

用 dispatch 表:
  type_table[type].reader(...)   // 一行查表，永不加分支
  // 新增类型 → 表里加一行 → 调用代码不动
```

> **术语：OCP 开闭原则（Open-Closed Principle）**
> "对扩展开放，对修改关闭"。新增类型只在 dispatch 表追加数据，调用代码不用改。
> 这是软件工程五大原则（SOLID）中的 O，也是 Linux 内核 `file_operations` 的设计哲学。
>
> **三种方案对比**：
>
> | | dispatch 表 | switch-case | if-else |
> |--|:--:|:--:|:--:|
> | 新增类型 | 表加一行 | 每个 switch 加 case | 每个 if 加 else if |
> | 查表效率 | O(1) 数组下标 | O(1) 跳转 | O(n) 逐个比较 |
> | 可传参 | 表可作为参数传递 | 不能 | 不能 |
> | 调试断点 | 难（不知道调到哪个函数） | 易 | 易 |
> | 可读性 | 需翻表定义 | 一目了然 | 一目了然 |
> | 类型安全 | 编译期不检查函数签名 | — | — |
>
> dispatch 表的劣势集中在可调试性和类型安全上——它是数据驱动的设计，代价是牺牲了 IDE 和编译器的辅助。但当类型数量超过一定阈值（bfile 有 11 种），数据驱动方案的扩展成本远低于控制流方案。

## 三、函数指针与 Dispatch Table

这样，就引出了函数指针，与储存它的 Dispatch Table（调度表）。通过它，我们可以用索引值直接路由，对应到相应函数。考虑到数字索引不明晰，我们采用了枚举值来做入口，以数据类型命名的枚举值，既清晰，又实现了索引功能。

**Dispatch Table 定义**：一个数据结构（通常是数组），将标识符映射到函数指针，使调用者无需条件分支即可选择正确的执行路径。

三要素：
```
1. Key（标识符）   → 枚举值 / 整数 / 字符
2. Lookup（查找）  → 数组 / 结构体数组 / 哈希表
3. Target（目标）  → 函数指针
```

由于我们需要有文件类型转换的操作，需要了解文件类型和读取字节大小，于是使用了一个结构体，将三者关联起来。所以实际上，我们的 Dispatch Table 是一个结构体数组——每个槽位包含 `name`（打印用）、`size`（read 多少字节）、`reader`（调到哪个函数）。这个表只需要查一次，就可以获得所有的信息。

```
type_table[TYPE_UINT32] = {
    .name   = "uint32_t",    // 打印类型名
    .size   = 4,              // read(fd, buf, 4)
    .reader = read_uint32,    // 解读 4 字节 → uint32_t
};

查一次表 → 三种信息全有了
```

## 四、两大结构体与枚举下标机制

此外，为了方便管理我们对命令的读取，我们也设置了专门的结构体，来储存枚举值、类型名和"数组字段元素数量"（如果有）。

```
type_info_t  — 编译期静态表，只一份，存"类型元信息 + 处理函数"
                  name / size / reader

field_def_t  — 运行期动态生成，parse.c 从格式串拆出来的字段列表
                  type / name / array_size

连接方式：fields[j].type = TYPE_UINT32
         → type_table[fields[j].type] = type_table[2]
         → {"uint32_t", 4, read_uint32}
```

两大结构体之间通过**枚举值 = 数组下标**这个机制完成了连通。枚举定义、表初始化、parse 遍历、main 查表——四者通过同一个整数串联。这个整数在枚举里叫 `TYPE_UINT32`，在数组里叫下标 2，实际上是一个东西。

> **引申：C 语言枚举的本质**
> `typedef enum { TYPE_UINT8=0, TYPE_UINT16=1, ... } field_type_t;`
> 枚举就是给 `int` 常量起名字。编译器自动从 0 递增赋值。GDB 调试时显示
> `TYPE_UINT32` 而不是 `2`——比 `#define` 强在类型安全和可调试性。

## 五、大小端转换

你的 x86 机器是小端（Cortex-M 也是），但固件文件可能是大端写的。

```
大小端只对 >1 字节的类型有意义:

  uint8_t  → 1 字节  → 无需 swap
  uint16_t → 2 字节  → swap16 交换高低字节
  uint32_t → 4 字节  → swap32 交换 4 个字节的排列
  uint64_t → 8 字节  → swap64（本项目未实现）

swap16: 0x1234 → 0x3412
swap32: 0x12345678 → 0x78563412
```

bfile 默认小端，用户用 `-e big` 显式指定大端。读取流程：`memcpy` 拼成整数 → 如果是大端 → `swap` 纠正 → 输出。

## 六、main.c 的编排复杂度

这是本项目被低估的难点。main.c 需要同时追踪：

```
for 每个字段:
  ├─ 维护 offset_current（打印偏移量，每次读完要前进）
  ├─ 区分 scalar / char数组 / 普通数组 三条分支
  ├─ 查 dispatch 表取 reader
  ├─ 每次 reader 调用前清零 value（uint64_t 复用残留问题）
  ├─ 格式化输出对齐
  └─ reader=NULL 时仍需前进 offset_current（字节已经读过了）
```

和 fdb 的直线型 main（每命令一个分支，调一个函数）不同，bfile 的 main 是**层叠结构**——循环套分支，分支里管状态，状态影响下一次循环。

**踩坑记录：**
- `offset_current` 在打印前被累加了 → 输出了"结束偏移"而不是"起始偏移"
- reader=NULL 分支没前进 offset_current → 后续字段偏移全错
- `uint64_t value` 复用未清零 → `read_uint8` 只改低 1 字节，高位残留上次结果
- char 数组用逐字节循环打印 `0x55` `0x42` 而不是当字符串输出 `"UBoot"`

## 七、实际输出的本质

我们实际输出的内容，只有 value 是源自于文件的——它是转译后的结果。其余的 offset、type 名、field 名，都是从我们命令中拆分出来的包装信息。

```
0x0000    uint32_t     magic       0x12345678
──────    ────────     ──────      ──────────
offset    type名       field名     file里的字节转译
程序维护  从格式串来    从格式串来   从文件来
```

解析器的价值：**用元数据包裹原始数据，让人看懂。**

---

## 八、总结与定位

本项目具有特殊性：其功能极其简单——把二进制文件按类型定义翻译成可读的值。但却需要考虑很多方面，也需要用到函数指针、调度表、大小端的知识点，对于 C 语言功底考验很深。

我们更多应该去学习它的代码思路，以及如何串联代码知识和实际问题（比如怎么想到函数指针、调度表、枚举类型、结构体？）。这是一个引子般的项目，更多的 C 语言功底考验，需要后续去学习。

| 涉及概念 | 掌握程度 | 后续强化 |
|----------|:--:|------|

| Dispatch Table | 初次接触 | Phase 3 file_operations |

| 结构体对齐 | 已理解 | Phase 2 container_of |
| 枚举 + 数组下标映射 | 已掌握 | 贯穿全路线 |

**代码量**：~300 行（含头文件声明），三个模块，11 种类型支持。
